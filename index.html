A Short history of JavaScript frameworks

AKA Why you should probably use a framework if you're doing client-side development

In the beginning...

JavaScript created by Brendon Eich for Netscape

In the beginning...

Each browser had their own version of a JavaScript engine

Yikes!

Over the following months and years others copied... sort of. Each had their own JavaScript engine with their own API, and one even tried a second scripting language...

The Result?

Chaos!

Ok not really...

But trying to use JavaScript was a pain...

Since there was an inconsistent JavaScript API for the DOM (or inconsistent behavior), using it was painful since developers couldn't trust that their code would run the same across all browsers

But then two things happened...

jquery and ajax solved a couple of key issues

ajax

  • Asynchronous JavaScript And XML
  • Brings data to the client-side asynchronously
  • Uses the XmlHttpRequest object (thanks Microsoft!)
The async part is key. It provided a way to fetch data without locking the UI while the request completed. Otherwise fetching data from the server via script would be problematic to the point of being unusable

jquery

  • Finally! a common DOM API
  • Whee! look at us manipulate the DOM
jquery made client-side development tolerable, even enjoyable. Developers no longer had to do the whole "if IE do X, if firefox do Y". manipulating the DOM was a breeze and so was developing rich, responsive apps.

The Result?

All the pieces exist to create rich, interactive, data-driven web applictions

The problem?

We had all the pieces, and no structure

We have a way to get data, we have a way to update the DOM wihtout server round-trips that works (mostly) across all browsers. But these can be thrown into a single file like a big bowl of sphagetti. or you can make a browser load 100s of individual files.

Frameworks!

Let the framework wars begin!

Frameworks are not for the browser, they're for the developer!

frameworks are always less effecient than native javascript. they exist to make the developer more effecient, and to keep the developer from hating life

Why should I use a framework?

You'll get...

  • Modularization (classes, components, etc)
  • Template-based rendering (html)
  • Guidance on code organization
  • Routing (not all, but most)
  • A way to interact with an API
  • Some security (XSS, script injection, etc)
  • Rich user experience
You'll also get the added benefit of leveraging a lot of really smart people's experience and not learn the hard way by writing 'low-level' JavaScript

Anatomy of a single page app (SPA)

High-level concept

  • Single HTML page (hence the name...)
  • Application logic managed by JavaScript
  • Navigation results in HTML fragments being added to/removed from DOM, rather than going to a new endpoint
  • Application state is contained entirely on the client. keeps server stateless!
  • Interaction with server is primarily AJAX calls

Pieces of a SPA

  • Framework - see previous slides :). May have influence over the other pieces of your app
  • Transpiler - Transforms ES6 or higher order languages to lower-level ES5 that can run across browsers (optional)
  • Testing framework - Yes, even JavaScript should be unit tested!
  • Module loader - loads all of the files you're adding for your framework
  • Bundler - concatenate/minify all the things! (many module loaders also serve as a bundler)
  • Linter - Keeps an eye on your code and highlights issues
  • Build tool - glues all the tools together. Some module loaders/bundlers can also serve as a build tool
